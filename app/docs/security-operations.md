# Secrets, retention, and operational policy

Generate `PAYLOAD_SECRET`, optional `AUTH_FLOW_ENCRYPTION_KEY`, `CRON_SECRET`, and webhook secrets independently with a cryptographically secure generator. Store environment secrets only in the target environment's secret manager. The Xero accounting client secret is configured by an owner/admin and stored as an AES-256-GCM envelope in the hidden singleton Payload record; the application derives separate configuration and token keys from `PAYLOAD_SECRET` with purpose-specific HKDF contexts. Xero identity secrets, Resend keys, and Mongo credentials remain separate per environment and least privilege.

Treat `PAYLOAD_SECRET` as a persistent root key. A planned change requires a migration that can read the old purpose-derived keys and re-encrypt with the new ones; without that migration the saved accounting client secret and tokens are unreadable. For an accounting client/token compromise, pause processing, revoke the Xero connection/secret remotely, clear usable local grants through protected disconnect, enter the replacement client credentials in `/app/settings/xero`, reconnect the pinned tenant, and reconcile unresolved exports. Rotate cron/webhook secrets at both caller/provider and application with an overlap only when the provider supports it.

Audit events are append-only and retained for at least the statutory/business invoice period; application logs should use a shorter documented retention and must not contain customer descriptions or credentials. Expired OAuth flow/session/invitation artifacts use TTL cleanup. Archived users/customers/projects remain referenced by historical time/export/audit snapshots.

Offboarding deactivates the local user, revokes Payload and external sessions, revokes the identity link, and preserves minimal identity/audit history. It never disconnects the business accounting grant. Before an accounting authorizer leaves, nominate another owner/admin and complete the guarded same-tenant handover.

Atlas alerts must cover connection saturation, storage, replication, and availability. Operational alerts must cover login/OIDC failures, link collisions, accounting loss/token failure, authorizer departure, invalid webhooks, 429 exhaustion, stuck/reconciling exports, manual review, and failed backups. Configure recipients and test alerts in staging; repository code cannot provision recipients.
